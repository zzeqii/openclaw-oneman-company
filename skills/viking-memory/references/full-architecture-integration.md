# 完整分层记忆架构：mem9+db9+OpenViking+QMD

现在我们把四个组件整合到上面设计的五层解决方案中，形成完整可落地的架构。

---

## 组件定位

每个组件在架构中承担明确职责，没有重叠：

| 组件 | 层级定位 | 核心职责 |
|------|----------|----------|
| **mem9** | 语义记忆层 + L3存储 | 长期语义记忆存储、向量搜索、一致性索引 |
| **db9** | 监控分析层 + 结构化存储 | 操作日志、性能指标、审计追踪、访问频率统计 |
| **OpenViking** | 代码上下文层 | 代码仓库语义索引、代码片段检索 |
| **QMD** | 文档检索层 | Markdown文档混合搜索（BM25+向量） |

## 各层级详细设计

### 1️⃣ 语义记忆层 (mem9)

**存储内容**：
- 用户偏好和习惯
- 重要决策和结论
- 项目关键信息摘要
- 文件元数据和时间戳索引（用于一致性检查）

**核心功能**：
- 向量语义搜索
- 混合关键词+向量检索
- 一致性验证索引
- 自动TTL过期清理

**存储位置**：
```
~/.local/openclaw/mem9/
```

**解决的问题**：
- ✅ 一致性问题（文件时间戳索引）
- ✅ 检索失效问题（语义搜索）
- ✅ 性能衰减问题（内置缓存）
- ✅ 安全性问题（本地存储）

---

### 2️⃣ 监控分析层 (db9)

**存储内容**：
- `memory_operations` - 所有记忆操作日志
- `performance_metrics` - 查询延迟、命中率
- `consistency_checks` - 一致性检查结果
- `access_frequency` - 记忆访问频率统计（用于冷热迁移）
- `error_logs` - 错误追踪

**核心功能**：
- 全链路审计追踪
- 性能指标监控
- 冷热判断依据
- 健康检查报告

**存储位置**：
```
本地PostgreSQL (db9-server)
数据库: openclaw_memory
```

**解决的问题**：
- ✅ 可观测性问题（完整日志）
- ✅ 性能衰减问题（访问频率统计支持冷热迁移）
- ✅ 一致性问题（检查结果记录，便于追踪）
- ✅ 安全性问题（审计日志支持异常检测）

---

### 3️⃣ 代码上下文层 (OpenViking)

**存储内容**：
- 代码仓库语义索引
- 函数/类/文件嵌入向量
- 代码依赖关系索引

**核心功能**：
- 基于问题检索相关代码
- 快速找到代码上下文
- 增量更新索引

**存储位置**：
```
~/.local/openclaw/openviking/
```

**解决的问题**：
- ✅ 检索失效问题（代码专用语义索引）
- ✅ 性能衰减问题（增量更新，不需要全量重建）
- ✅ 安全性问题（本地索引，代码不上传第三方）

---

### 4️⃣ 文档检索层 (QMD)

**存储内容**：
- Markdown文档BM25索引
- 文档片段向量索引
- 文档元数据

**核心功能**：
- 关键词+向量混合搜索
- 支持大规模Markdown文件
- 增量更新

**存储位置**：
```
~/.local/openclaw/qmd/index.db
```

**解决的问题**：
- ✅ 检索失效问题（混合搜索比单一向量更准确）
- ✅ 性能衰减问题（SQLite存储，高效稳定）
- ✅ 一致性问题（文档变更自动更新索引）

## 工作流程设计

### 会话启动流程

```
1. 读取 AGENTS.md
2. 读取 SOUL.md
3. 读取 USER.md
4. 🚨 mem9一致性检查
   → 对比文件mtime与索引
   → 发现不一致 → 自动更新索引
   → 严重不一致 → 报告用户确认
5. 读取今日+昨日记忆文件
6. 如果是主会话 → 读取 MEMORY.md
7. 预热L1缓存（当前项目相关记忆）
8. 就绪，接受用户请求
```

### 记忆存储流程

```
用户提供信息 → 判断内容类型
   ├─ 语义记忆/用户偏好 → mem9存储
   ├─ Markdown文档 → QMD索引 + mem9摘要
   ├─ 代码文件 → OpenViking索引
   └─ 所有操作 → db9日志记录
```

### 记忆检索流程

```
用户查询 → 解析查询类型
   ├─ 并行查询相关层级：
   │  ├─ 语义问题 → mem9搜索
   │  ├─ 文档查找 → QMD搜索
   │  ├─ 代码问题 → OpenViking搜索
   │  └─ 历史操作 → db9查询
   ├─ 合并所有结果
   ├─ 按相关性加权排序
   ├─ 去重
   ├─ 记录查询到db9
   └─ 返回Top-N结果给用户
```

### 一致性检查流程

```
触发 → 遍历所有索引文件
   ├─ 获取实际文件mtime
   ├─ 对比索引中记录的mtime
   ├─ 如果不同：
   │  ├─ 文件被修改 → 更新对应索引
   │  ├─ 文件被删除 → 从索引移除
   │  └─ 新文件 → 添加到索引
   ├─ 记录检查结果到db9
   └─ 生成一致性报告
```

### 冷热迁移流程

（每日自动运行）

```
从db9获取访问频率统计
   ├─ 高频访问 → 提升到L2缓存
   ├─ 中频 → 保持在原层级
   └─ 低频 (>90天未访问) → 迁移到冷存储
更新mem9缓存层级
记录迁移日志到db9
```

## 配置设计

### 环境变量 `.env.memory`

```env
# mem9配置
MEM9_DATA_DIR=~/.local/openclaw/mem9
MEM9_EMBEDDING_MODEL=text-embedding-3-small
MEM9_CACHE_SIZE=1000

# db9配置
DB9_CONNECTION_STRING=postgresql://localhost:5432/openclaw_memory
DB9_AUTO_START=true
DB9_ENABLE_METRICS=true

# OpenViking配置
OPENVIKING_INDEX_DIR=~/.local/openclaw/openviking
OPENVIKING_ENABLED=true
OPENVIKING_CHUNK_SIZE=512

# QMD配置
QMD_INDEX_PATH=~/.local/openclaw/qmd/index.db
QMD_EMBEDDING_ENABLED=true
QMD_BM25_ENABLED=true
```

## 优势总结

### 对比传统架构

| 方面 | 传统Git记忆架构 | 本分层架构 |
|------|----------------|------------|
| 一致性 | 依赖手动维护，经常不一致 | 自动检查自动修复 |
| 检索能力 | 只支持关键词搜索 | 混合语义搜索，准确率高 |
| 性能 | 记忆增长后线性变慢 | 冷热分层，性能稳定 |
| 安全性 | Git存储所有，易泄漏 | 敏感数据本地隔离 |
| 可观测性 | 无日志，黑盒 | 全链路监控，问题可追踪 |

### 核心创新点

1. **问题驱动设计**：从分类问题出发，每个问题对应一个层级解决方案
2. **职责清晰分离**：四个组件各司其职，没有重叠，易于维护
3. **完全本地优先**：所有数据都在用户本地，不依赖第三方云服务
4. **Git责任分离**：Git只存代码和文本日记，不存索引和敏感数据
5. **自动化运维**：一致性检查、冷热迁移、健康检查全自动化

## 扩展能力

架构设计预留了扩展点：

- 支持添加新的索引层级（比如图片检索、音频检索）
- 支持替换底层存储引擎（比如用Chroma替代TiDB）
- 支持云端同步（用户自愿开启，默认关闭）
- 支持多设备同步通过用户自己的云存储
