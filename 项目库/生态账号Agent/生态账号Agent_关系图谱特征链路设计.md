# 生态账号Agent 关系图谱特征链路设计
## 核心目标：通过关系图谱特征识别异常账号，输出风险判定及解释
---
## 一、特征层级链路图
```
原始特征（原子层） → 加工特征（计算层） → 最终特征（应用层）
     ↓                   ↓                    ↓
  原始数据存储        图数据库存储        特征服务调用
```
---
## 二、各层级详细说明
### 🔹 第一层：原始特征（原子特征）
#### 定义：从业务系统直接采集的未加工原始数据，是关系图谱的基础节点和边
| 原始特征类别 | 具体内容 | 数据来源 | 存储方式 |
| --- | --- | --- | --- |
| IP关联 | 账号登录IP、操作IP、注册IP | 登录日志、操作日志 | 宽表存储（Hive/ClickHouse），按账号/IP维度分区 |
| 设备关联 | 设备ID、设备指纹、MAC地址 | 设备采集SDK | 宽表存储，按账号/设备维度分区 |
| 付款账号关联 | 支付账号、银行卡号、第三方支付ID | 支付系统 | 加密存储，脱敏后使用 |
| 身份证关联 | 身份证号、实名认证信息 | 实名认证系统 | 加密存储，仅用于关联计算 |
| 社交关联 | 直播间互动、群聊成员、评论@关系、关注/粉丝关系 | 内容平台、社交系统 | 图边数据，原始存储 |
| 时空聚集 | 地理位置、操作时间、线下场景关联 | LBS数据、操作日志 | 时空数据库存储 |
| 内容特征 | 内容画风相似度、文案重合度、视频指纹 | 内容审核系统 | 向量数据库存储 |
| 行为突变 | 登录设备变化、投稿时段变化、内容垂类变化、资料修改频率 | 用户行为系统 | 时序数据库存储 |
---
### 🔹 第二层：加工特征（中间特征）
#### 定义：基于原始特征通过图计算、规则计算生成的中间特征，描述账号之间的关联关系
| 加工特征类别 | 计算逻辑 | 加工方法 | 存储方式 |
| --- | --- | --- | --- |
| 关系强度 | 账号之间的关联紧密程度，关联维度越多、频次越高，强度越高 | 加权求和：IP关联(0.3)+设备关联(0.3)+付款关联(0.4)等 | 图数据库（Neo4j/JanusGraph），作为边的属性存储 |
| 团伙类型 | 账号所属团伙的类别：<br>• 同人运营：多账号同一人操作<br>• 水军利益链：批量刷量/控评团伙<br>• 教学师徒：MCN/导师带徒模式<br>• 账号买卖：交易流转关系 | 规则引擎+图聚类算法（Louvain/Label Propagation） | 图数据库，作为节点属性存储 |
| 团伙规模 | 团伙包含的账号数量、关联的核心节点数量 | 图遍历统计 | 图数据库，作为节点属性存储 |
| 关联路径 | 两个账号之间的最短关联路径、关联维度 | 最短路径算法（Dijkstra/BFS） | 实时计算，不存储 |
| 异常关联度 | 账号与已知黑产账号的关联程度 | 标签传播算法 | 图数据库，作为节点属性存储 |
---
### 🔹 第三层：最终特征（应用特征）
#### 定义：直接输入账号风险模型的特征，用于判定账号是否异常
| 最终特征 | 特征含义 | 调用方式 |
| --- | --- | --- |
| 团伙风险标签 | 账号所属团伙类型，无团伙则为"正常" | 特征服务接口，传入账号ID直接返回 |
| 团伙风险评分 | 0-100分，分数越高风险越高 | 特征服务接口，实时计算返回 |
| 关联异常账号数 | 该账号关联的已知异常账号数量 | 特征服务接口，实时统计返回 |
| 最高风险关联路径 | 该账号与最高风险账号的关联路径详情 | 特征服务接口，实时计算返回 |
| 关系异常度 | 账号关联关系的异常程度，是否存在异常聚集 | 特征服务接口，离线计算T+1更新 |
---
## 三、存储与调用架构
### 1. 存储架构
```
原始数据层 → 计算层 → 特征服务层
   ↓           ↓         ↓
Hive/ClickHouse → 图数据库Neo4j → 特征服务API
向量数据库Milvus ↗
时序数据库InfluxDB ↗
```
### 2. 调用流程
1. 账号风险查询请求到达
2. 特征服务从图数据库查询该账号的所有加工特征
3. 结合实时计算的关联路径、异常度等特征
4. 输出最终特征给风险模型
5. 模型返回风险判定结果+可解释性说明（关联路径、团伙类型等）
